# Phase 16 Execution Plan — Error taxonomy + CLI boundary

> Narrow error-handling centralization for the single-user `startup-radar` tool. Introduce a three-class exception taxonomy in a new `startup_radar/errors.py` module, rebase the sole existing custom exception (`ConfigError` in `config/loader.py:17`) onto it, and wire a `--debug`-gated Typer error boundary so every CLI command renders a single-line failure for `StartupRadarError` subclasses and a full traceback only when the user opts in. **Explicitly NOT** a sprawling hierarchy — no `HTTPError` wrapper (`httpx.HTTPError` already exists and is caught by `_retry`), no `StorageError` (nothing would catch it distinctly), no `cli/io.py` `ok()/fail()/section()` print helpers (user-facing terminal output is already sanctioned under `.claude/rules/observability.md`). Abstractions earn their keep by being *caught*, not by being raised.

## Phase summary

- **One tiny errors module: `startup_radar/errors.py` (~15 LOC).** Exposes exactly three classes. `StartupRadarError(Exception)` is the base — one-line docstring "Base for all known, user-facing failure modes raised by startup_radar." `ConfigError(StartupRadarError)` is rebased from `startup_radar/config/loader.py:17` and keeps its existing docstring verbatim ("Raised when config.yaml is missing, unparseable, or fails schema validation."). `SourceError(StartupRadarError)` is new — available for any source that wants to abort the run rather than return `[]`. `__all__ = ["StartupRadarError", "ConfigError", "SourceError"]`. No other classes, no metaclass theatrics, no `.code` attribute nobody reads.
- **Why these three and nothing else.** `ConfigError` already exists and already has a concrete catch site (`cli._doctor` line ~392). `SourceError` earns its keep because the pipeline's `try/except/finally` → `storage.record_run(..., error=str(exc))` is the concrete catch site that differentiates "source said no" from "bug in our code" once any future source wants to raise. `StartupRadarError` is the base the CLI boundary filters on — "known failure" vs "bug we should surface loudly." Every other wrapper the original draft considered (`HTTPError`, `StorageError`, `RetryExhausted`, `DedupError`, etc.) has no concrete catch site that would react differently from a generic exception — declining to add them is the win, not a gap.
- **Back-compat: `ConfigError` stays importable from the loader.** `startup_radar/config/loader.py` drops its local `class ConfigError(Exception):` and adds `from startup_radar.errors import ConfigError` at module top. Any existing `from startup_radar.config.loader import ConfigError` (and the re-export via `from startup_radar.config import ConfigError` in `config/__init__.py`, if present) still resolves — the import statement re-binds the name at module scope. Zero churn for existing call sites.
- **Top-level Typer error boundary — `@_catch_errors` decorator.** A single ~10-LOC decorator defined in `cli.py`:
  - Wraps every `@app.command()` body in `try: … except StartupRadarError as e: log.error("cli.error", command=func.__name__, error=str(e), exc_info=<debug>) ; typer.echo(f"Error: {e}", err=True) ; raise typer.Exit(1)`.
  - `typer.Exit` passes through untouched (the commands already use it as a normal control flow).
  - Generic `Exception` **does not** get swallowed — bugs keep surfacing with full tracebacks, matching the `.claude/rules/observability.md` §4 rule ("never `except Exception:` without re-raising or logging at error with traceback").
  - Applied to all six commands: `run`, `serve`, `deepdive`, `doctor`, `status`, `backup`.
- **`--debug` opt-in for tracebacks.** A new `debug: bool = typer.Option(False, "--debug", help="Show full tracebacks on error.")` on the top-level `@app.callback`. The callback stashes it on `ctx.obj` (a tiny `@dataclass class CLIState: debug: bool = False` — same pattern Phase 14 sketched for `--config`, so if both ship together the two fields live on one dataclass). The decorator reads `ctx.obj.debug` and threads it into the `log.error(..., exc_info=debug)` kwarg so structlog emits the traceback into the log record when the user opts in. Default: one-line error, no traceback. With `--debug`: traceback in logs + traceback in terminal.
- **Source contract: docstring nudge, zero source rewrites.** `startup_radar/sources/base.py` gets one added paragraph on `fetch()`: "Raise `SourceError` on a non-transient failure the caller should surface to the user. Transient failures (network blips, 5xx, timeouts) are retried by `startup_radar.sources._retry` and ultimately swallowed into `[]` per the observability rules." The four existing sources (`rss`, `hackernews`, `sec_edgar`, `gmail`) are **not** rewritten in this phase — their current `try/except → log.warning → return []` pattern is still valid per `.claude/rules/observability.md` line 16. `SourceError` is available for future sources that want stricter semantics; changing the existing four would be pure churn.
- **Audit greps (read-only guardrails, baked into exit criteria).** Phase 16 enforces the observability rules with a handful of greps that must come back clean. If any return violations, fix them inside this phase. The audits:
  - `grep -rn "raise Exception(" startup_radar/` — zero matches.
  - `grep -rn "except Exception:" startup_radar/` — each remaining match must be paired with `log.error(..., exc_info=True)` + re-raise, or be inside a narrowly-scoped fallback (e.g. `cli._source_health` where swallowing is the documented contract). Review every hit; fix the sloppy ones.
  - `grep -rn "os.getenv\|os.environ" startup_radar/` — only `startup_radar/config/secrets.py` and (if Phase 14 has landed) `startup_radar/config/loader.py`'s `STARTUP_RADAR_CONFIG_PATH` subprocess hand-off.
  - `grep -rn "logging.basicConfig" startup_radar/` — zero matches outside `observability/logging.py`.
  - `grep -rn "^print(" startup_radar/` — zero outside `cli.py` and `research/deepdive.py`.
- **No new lint rules.** Ruff doesn't have a great native expression for "`StartupRadarError` subclasses only", and a custom plugin is over-engineering for a single-user tool. The audit greps are run by hand during phase execution and documented in the exit criteria; future drift is caught at review time.
- **Logging stays structlog.** The error boundary uses `get_logger(__name__).error("cli.error", command=..., error=..., exc_info=debug)` — structured kwargs per `.claude/rules/observability.md` line 8. No formatted strings, no `extra={}`.

## Out of scope

- **`HTTPError` wrapper class.** `httpx.HTTPError` already exists, `_retry` already catches it, and our sources already translate persistent HTTP failures into `log.warning → return []`. Wrapping just to rename is churn with no call site reacting differently.
- **`StorageError` wrapper class.** No concrete catch site would treat it differently from a generic exception today. Add it only if/when a caller wants to react to storage failures distinctly (e.g. backup-on-corruption flow) — not this phase.
- **`cli/io.py` with `ok()`/`fail()`/`section()` helpers.** The `print()` calls in `cli.py` and `research/deepdive.py` are user-facing terminal output, already sanctioned by `.claude/rules/observability.md` line 10. Wrapping them in helper functions is aesthetic DRY that buys nothing when there's one CLI and one research module.
- **Rewriting the four existing sources to `raise SourceError(...)`.** Their current `except → log.warning → return []` pattern is valid per the observability rules. Changing them would be churn without a concrete catch site that cares about the distinction yet.
- **`RetryExhausted` / `RateLimited` / `CircuitOpen` classes.** `_retry` today logs the final failure and re-raises the underlying exception; the retry-exhaustion signal is already carried by the exception type the caller sees. Circuit-breaker is §4.4 deferred in `docs/PRODUCTION_REFACTOR_PLAN.md`.
- **Replacing the `get_logger` / structlog pipeline.** Centralized in Phase 11; this phase uses it, doesn't touch it.
- **Replacing the `secrets()` / pydantic-settings pipeline.** Centralized in Phase 12; untouched.
- **`AppConfig` / `load_config` restructure.** Centralized in Phases 5 + 12; untouched.
- **A ruff plugin or custom linter to auto-enforce the audit greps.** The greps run once during phase execution and again in the exit-criteria checklist; a plugin is over-engineering for a single-user tool.
- **Sentry / telemetry hookup.** Dropped from the overall plan per §0a.

## Tests

1. `tests/unit/test_errors.py` — the three cases listed in the file-changes table. Covers inheritance, back-compat re-export, message round-trip, and the CLI boundary (`--debug` off → no traceback; `--debug` on → traceback present).
2. `tests/unit/test_cli_doctor.py` (existing) — verify it still passes; `_doctor` imports `ConfigError` from `startup_radar.config` (which re-exports from the loader, which now re-exports from `errors.py`). No code change to the test expected.
3. `tests/unit/test_cli_status.py` and `tests/unit/test_cli_backup.py` (existing) — verify they still pass after the `@_catch_errors` decorator is applied to `status` and `backup`. The decorator should be transparent on the happy path.
4. **Audit-grep gate** — run the five greps listed in the phase summary locally and in the exit criteria. Any violations get fixed in this phase before commit.
5. **Manual — broken config.** Edit `config.yaml` to produce a `ConfigError` (e.g. break YAML syntax), run `uv run startup-radar status` — expect exit 1 with a single-line `Error: ...` and no traceback. Then run `uv run startup-radar --debug status` — expect exit 1 with the traceback visible in the log output.
6. `make ci` green: ruff + format-check + mypy + full pytest, coverage maintained.
