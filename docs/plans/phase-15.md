# Phase 15 Execution Plan — MkDocs Material docs site + GitHub Pages publish

> Close the last remaining Tier 4 polish item (`docs/PRODUCTION_REFACTOR_PLAN.md` execution table row 17: "MkDocs site (optional)"). Stand up a contributor-facing MkDocs Material site under the existing `docs/` tree, wire the live markdown already on disk (`PRODUCTION_REFACTOR_PLAN.md`, `CRITIQUE_APPENDIX.md`, `AUDIT_FINDINGS.md`, every `docs/plans/phase-*.md`, `docs/ops/data-branch.md`) into an explicit `nav:`, author three new prose pages (architecture, contributing, adding-a-source), and publish to GitHub Pages on every push to `main` that touches `docs/`. **Explicitly NOT** a `mkdocstrings`-backed API reference, **NOT** a versioned docs site via `mike`, **NOT** a custom theme / logo / analytics pass — those are either non-goals for a 5k-LOC single-user tool or later, separately-scoped phases.

## Phase summary

- **Stack: MkDocs + Material theme + `mkdocs-material` + `pymdown-extensions`.** No Sphinx, no Docusaurus. MkDocs is Python-native (fits the existing `uv` workflow), Material is the most popular theme and needs zero CSS/JS tinkering, and `pymdown-extensions` (pulled in transitively by `mkdocs-material`) ships syntax highlighting, admonitions, tabbed code blocks, and `superfences` (Mermaid support) out of the box. All new deps land in a fresh `[project.optional-dependencies]` entry named `docs` — deliberately **not** a `[tool.uv].dev-dependencies` entry, so users who want to build docs locally don't need the full dev kit and `make ci` doesn't widen its install surface.
- **Why MkDocs Material over Sphinx / Docusaurus.** (1) Python-native, fits `uv sync`. (2) `docs_dir` defaults to `docs/` — which is exactly where our existing markdown lives, so no file relocation. (3) Material's `navigation.instant` + `search` + `content.code.copy` give a GitHub Pages site that looks right without bespoke theming. (4) `mkdocs-include-markdown-plugin` lets us embed snippets from `.claude/CLAUDE.md` and `README.md` into docs pages so the source of truth stays where it's already maintained.
- **Autolink-friendly content structure.** Nav sections: **Home** (short `index.md`, embeds README hero via include-markdown), **Getting Started** (thin page deep-linking into the existing README sections), **Architecture** (new prose page), **Contributing** (new prose + conventions), **Sources** (author guide + one one-pager per live source), **Operations** (data-branch, backups, scheduling), **Reasoning trail** (PRODUCTION_REFACTOR_PLAN, CRITIQUE_APPENDIX, AUDIT_FINDINGS, a phases index). No auto-generated API reference — this is a 5k-LOC single-user tool; a hand-written `ARCHITECTURE.md` is higher signal than `mkdocstrings` dumps.
- **`mkdocs-include-markdown-plugin` used to avoid duplication.** The "Core invariants" + "Gotchas" blocks in `.claude/CLAUDE.md` get embedded into `docs/contributing/conventions.md` via `\{% include-markdown ".claude/CLAUDE.md" start="<!-- invariants:start -->" end="<!-- invariants:end -->" %\}` markers (tiny marker-comment edit to CLAUDE.md, does not affect how Claude reads it). The README hero is embedded into `docs/index.md` the same way. CLAUDE.md stays canonical — the docs site is a read-through view.
- **Only three prose docs authored from scratch in this phase.** (1) `docs/architecture.md` — ~200 lines: package tree (embedded from CLAUDE.md), data flow (config → sources → filters → storage → dashboard), migrator note pointing at `startup_radar/storage/migrations/`, retry + logging invariants, Mermaid diagram via `pymdownx.superfences`. Cross-links existing CRITIQUE_APPENDIX entries for "why not X" items (no alembic, no Postgres, no async). (2) `docs/contributing/index.md` — dev-env setup (`uv sync --all-extras`, `make ci`, `.claude/hooks/` flow), commit conventions, PR flow, the `/ship` skill, how to add a phase plan. (3) `docs/sources/adding-a-source.md` — concrete walkthrough of adding a new `Source` subclass: `sources/base.py` contract, `sources/registry.py` wiring, `filters.py` integration, vcrpy cassette recording flow, the `source-implementer` subagent, how to add the source to `doctor --network`. Pulls sample code from `sources/hackernews.py` (simplest live source).
- **Reuse existing markdown where possible.** `docs/PRODUCTION_REFACTOR_PLAN.md`, `docs/CRITIQUE_APPENDIX.md`, `docs/AUDIT_FINDINGS.md`, every `docs/plans/phase-*.md` — all exist, stay where they are, wired straight into `nav:` by relative path. `docs/ops/data-branch.md` is the **one** file this phase relocates: `git mv docs/ops/data-branch.md docs/operations/data-branch.md` so all operational docs live under one canonical nav section. One `grep -rn 'docs/ops/data-branch' docs/ .claude/` pass, rewrite the two or three references (`AGENTS.md`, `CLAUDE.md` gotcha, `PRODUCTION_REFACTOR_PLAN.md` Tier 0 #1 callout), done.
- **GH Pages publishing is opt-in via a new workflow.** `.github/workflows/docs.yml` triggered on `push: branches: [main]` with `paths: ['docs/**', 'mkdocs.yml', '.github/workflows/docs.yml']` and `workflow_dispatch`. Uses `actions/configure-pages@v5` + `actions/upload-pages-artifact@v3` + `actions/deploy-pages@v4`. Mirrors `ci.yml`'s `setup-python` + `astral-sh/setup-uv@v3` pattern. `uv sync --extra docs` → `uv run mkdocs build --strict` → upload `site/` → deploy. First run requires the operator to flip **Settings → Pages → Source** to "GitHub Actions" — called out as a one-time operator step at the top of the workflow file and in the phase-exit summary.
- **Strict build by default.** `mkdocs build --strict` in CI (and in `make docs`). Any broken internal link, any page in `docs/` not listed in `nav:`, any missing include-markdown anchor → build fails. This is cheap protection against doc bitrot; the `CRITIQUE_APPENDIX` §10 "low-cost guardrails" posture applies.
- **`make docs` and `make docs-serve` added.** `make docs` builds `site/` locally via `uv run mkdocs build --strict`. `make docs-serve` runs `uv run mkdocs serve` on port 8000. Both rely on the `docs` extra being installed (`uv sync --extra docs`). Neither is part of the `make ci` aggregate — docs build is its own axis.
- **`site/` is gitignored.** The build output never commits; GH Actions publishes the live site directly from the build artifact uploaded by `actions/upload-pages-artifact@v3`.
- **Internal link hygiene: relative links only.** Existing phase plans reference `CRITIQUE_APPENDIX.md` via relative paths; those continue to resolve under MkDocs because `docs_dir` defaults to `docs/`. Any absolute GitHub URLs currently in `docs/` are audited during this phase — grep target is `grep -rn 'github.com/.*/blob/' docs/` — and converted to relative links so `mkdocs --strict` can validate them.

## Out of scope

- **API reference generation via `mkdocstrings`.** 5k-LOC single-user tool; a hand-written `architecture.md` is higher signal than per-module docstring dumps. Revisit only if the `startup_radar/` package becomes an importable library for third parties.
- **Custom theme, custom CSS/JS, logo/favicon, hero illustrations.** Material's defaults are fine for a contributor-facing site.
- **Translations / i18n.** Single-user tool, single-language audience.
- **Search indexing beyond MkDocs' built-in `search` plugin.** No Algolia, no external search provider.
- **Versioned docs via `mike`.** Single-user tool, no need to surface "docs for v0.3.x vs v0.4.x".
- **Tutorial-style long-form content.** The existing README narrative covers onboarding; the docs site is contributor-focused.
- **Moving `.claude/CLAUDE.md` content into `docs/` as a source of truth.** CLAUDE.md stays canonical — docs site embeds it via `include-markdown`, does not fork it.
- **PDF export, offline bundles, ebook builds.**
- **Analytics (Plausible, GA, Umami).** No telemetry, consistent with the refactor-plan §0a adds ("Explicit no-telemetry statement").
- **Populating the dashboard screenshot on `docs/index.md`.** Left as a `TODO:` comment for a follow-up phase — doesn't block the build.
- **Dropping the `docs/ops/` directory completely.** Only `data-branch.md` currently lives there; after the `git mv` the directory is empty and can be `git rmdir`'d as a trivial follow-up (or left — empty dirs don't ship to the site since MkDocs only walks `docs/`).

## Tests

No new tests under `tests/` — docs is prose-only, no Python surface to cover. Verification is the strict build plus a handful of greppable checks:

1. `uv sync --extra docs` on a clean `.venv` installs the three new deps without resolver conflicts.
2. `make docs` (i.e. `uv run mkdocs build --strict`) exits 0 with zero warnings. Two warning classes that must be zero: `"Doc file '…' contains a link to '…' that is not found"` and `"A reference to '…' is included in the 'nav' but not found among documentation files."`
3. `make docs-serve` runs locally; all six top-level nav sections (Home, Getting Started, Architecture, Contributing, Sources, Operations, Reasoning trail) load without 404s and search works.
4. Internal cross-references resolve. Specifically: `docs/architecture.md` links to `docs/sources/adding-a-source.md`; `docs/contributing/conventions.md` successfully embeds the `.claude/CLAUDE.md` "Core invariants" block (strict-mode build would fail if the anchors don't match); `docs/reasoning/phases.md` lists all 15 existing phase plans (phases 1, 2, 3, 4, 5, 6, 7, 7.5, 8, 9, 10, 11, 12, 13, 15 — phase 14 reserved if applicable; otherwise the list reflects what `ls docs/plans/*.md` prints).
5. `docs/index.md` include-markdown pulls the README hero — strict mode fails if the `<!-- hero:start -->` / `<!-- hero:end -->` markers aren't present in `README.md`.
6. `grep -rn 'docs/ops/data-branch' .` returns zero matches after the `git mv` + reference rewrites.
7. `grep -rn 'github.com/.*/blob/' docs/` returns zero matches (or only matches inside explicit "external reference" callouts) — absolute GitHub URLs converted to relative links.
8. `.github/workflows/docs.yml` runs successfully on the first post-merge push to `main`. First run requires the operator to flip **Settings → Pages → Source** to "GitHub Actions" — called out in the phase-exit notes and in the workflow's file header.
