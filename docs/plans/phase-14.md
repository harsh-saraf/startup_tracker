# Phase 14 Execution Plan — Dockerfile + optional compose for single-user deploy

> Deliver the Tier 2 "Dockerfile — single image, optional" row (`docs/PRODUCTION_REFACTOR_PLAN.md` execution-order row 16) without overshooting the §0a override. Produce a reproducible, non-root, multi-stage image that boots the Streamlit dashboard by default and can also run the pipeline one-shot via `docker run … run`. Ship a small `docker-compose.yml` as an optional convenience wrapper only — per §0a "Drops/Demotes", compose is "only if someone else ever runs this" and **Postgres is not a service**. Image registry publishing, a `HEALTHCHECK`, and GHCR CI workflow are **explicitly deferred**.

## Phase summary

- **One explicit `--config PATH` knob, no env-var fallbacks.** The original draft layered `STARTUP_RADAR_CONFIG_PATH` / `STARTUP_RADAR_DB_PATH` env vars on top of the existing CWD-relative / package-relative config resolution — two sources of truth for "where does config live." Rejected. Phase 14 adds a single `--config PATH` Typer option on the top-level `@app.callback` in `startup_radar/cli.py`, threaded into the existing `load_config(path=...)` parameter (which already accepts an optional path — `startup_radar/config/loader.py:21`). The Dockerfile `CMD` bakes `--config /config/config.yaml` into the image. One explicit knob, conventional, no new resolution paths, no new env vars.
- **DB path stays in `config.yaml`.** Users override it via `output.sqlite.path: /data/startup_radar.db` in their mounted `config.yaml` — one knob the code already reads (`cfg.output.sqlite.path`). Zero Python changes for the DB path.
- **Multi-stage build keeps the runtime image lean.** Stage 1 (`builder`) is `python:3.11-slim` with `uv` copied in from the official `ghcr.io/astral-sh/uv:latest` image; it runs `uv sync --frozen --all-extras --no-dev` into `/app/.venv`. Stage 2 (`runtime`) is a fresh `python:3.11-slim` that only copies the resolved `/app/.venv` plus the package source — no uv, no build toolchain, no dev deps. This mirrors how `.github/workflows/ci.yml` already pins Python 3.11 and uses `uv sync --all-extras`, so local Docker and CI agree on the resolved graph.
- **`uv.lock` is authoritative; never `pip install`.** The builder stage must run `uv sync --frozen` (fail if the lockfile drifts) — consistent with the `.claude/CLAUDE.md` "never `pip install -r requirements.txt`" invariant. `--no-dev` keeps pytest/mypy/ruff out of the image; `--all-extras` pulls in the `[google]` extra so a user who mounts real credentials can use Gmail without rebuilding.
- **Non-root by construction.** A named `radar` user (UID 10001, GID 10001) is created in the runtime stage. `/app`, `/data`, and `/config` are `chown`'d before the `USER 10001:10001` switch. Closes the "container process runs as root" finding every security scanner flags and costs us nothing — SQLite doesn't care which UID opens the file as long as the mount's perms match.
- **All state lives on mounted volumes.** The image is stateless: DB at `/data/startup_radar.db` (user-set via config.yaml), config + secrets at `/config/{config.yaml,credentials.json,token.json,.env}`. Single `VOLUME ["/data","/config"]` declaration. `CMD` bakes in `--config /config/config.yaml` so `docker run -v …` "just works" without per-invocation flags.
- **Default `CMD` is the dashboard.** `CMD ["startup-radar", "--config", "/config/config.yaml", "serve", "--port", "8501", "--address", "0.0.0.0"]`, `EXPOSE 8501`. Most users run `docker run -p 8501:8501 …` and hit the dashboard — this matches the `.claude/CLAUDE.md` "dashboard is Streamlit on port 8501" invariant and the north-star DX from §0. Overriding to `docker run --rm startup-radar --config /config/config.yaml run` invokes the pipeline once; `… run --scheduled` is the cron-style one-shot path already wired in Phase 11.
- **Dashboard config path.** `startup-radar serve` spawns Streamlit via `subprocess`, and Streamlit re-imports `web/app.py` which calls `load_config()` *without* the CLI's `--config`. The serve command needs to propagate `--config` into the child Streamlit process — either via an env var that `web/app.py` reads, or by writing the path into a known location. Prefer the env-var route **scoped to serve only** (one var, one consumer, not a sprawling fallback): `STARTUP_RADAR_CONFIG_PATH` is set by `serve` when `--config` is passed, and `load_config()` checks it only if no explicit path is provided. This is one controlled hand-off, not a general-purpose fallback chain, and it only fires for the dashboard subprocess.
- **No baked secrets — enforced by `.dockerignore`.** The image must contain zero copies of `.env`, `credentials.json`, `token.json`, or a real `config.yaml`. `.dockerignore` mirrors `.gitignore` (backups/, *.db, .env, token.json, credentials.json, .venv, logs/, reports/) plus `.git`, `.github`, `docs`, `tests`, `reports`, `backups`, `*.md`. Only `config.example.yaml` is copied — as documentation for the mount layout, not a runtime default.
- **`docker-compose.yml` is an optional convenience, not load-bearing.** Per §0a "Drops/Demotes", compose is downgraded to "only if someone else ever runs this". Ship a single-service `radar:` compose file that bind-mounts `./data:/data` and `./config:/config` and exposes `8501`. **No Postgres service** (dropped per §0a; DB stays single-file SQLite per `.claude/CLAUDE.md`). No separate scheduler service — use `docker compose run --rm radar startup-radar --config /config/config.yaml run` for the one-shot path.
- **Image size target ≤250 MB compressed.** Budget: `python:3.11-slim` (~45 MB) + resolved `.venv` with pandas + streamlit + `[google]` (~150 MB) + our package source (~2 MB) leaves slack for certifi / tz data / etc. Soft target, not a contract; document if blown.
- **Multi-arch buildx, documented not automated.** Document `docker buildx build --platform linux/amd64,linux/arm64 -t startup-radar:phase-14 .` in the README's "Running with Docker" section. **Do not** add a GH Actions workflow to publish to GHCR — deferred (see "Out of scope").
- **No `HEALTHCHECK` directive.** Adding one requires `curl` in the image (bloat) or a custom Python one-liner hitting `/_stcore/health` for ~zero single-user value. Compose / orchestrators that care can add their own `healthcheck:` block. Punt explicitly.
- **Makefile gets three thin wrappers.** `docker-build`, `docker-run`, `docker-shell` — pure aliases over `docker build`/`docker run`/`docker run --entrypoint bash`. Keeps the `make`-first DX from existing targets (`make serve`, `make run`, `make ci`) consistent.

## Out of scope

- **`STARTUP_RADAR_CONFIG_PATH` / `STARTUP_RADAR_DB_PATH` env-var fallbacks.** Explicitly rejected — two sources of truth for config resolution is the mess we're avoiding. The one env var we do set (`STARTUP_RADAR_CONFIG_PATH` inside `cli.serve`) is a controlled subprocess hand-off, not a user-facing knob. Documented accordingly.
- **GHCR publishing workflow.** No `.github/workflows/docker-publish.yml`, no image signing, no SBOM generation. Registry publishing is a separate phase if anyone else ever runs this.
- **`HEALTHCHECK` directive.** Would require `curl` in the runtime image or a custom Python one-liner. Compose users who care can add a `healthcheck:` block in their own override.
- **`docker-compose` Postgres service.** Explicitly dropped per §0a.
- **PyPI release.** Out of scope; `pyproject.toml` already builds cleanly.
- **Distroless base.** Breaks Streamlit + pandas's C-ext expectations and forces a complex COPY matrix. `python:3.11-slim` is the right knob.
- **Windows containers.** Linux image works fine on Docker Desktop's WSL2 backend.
- **Image signing (cosign / sigstore) and SBOM emission.** Deferred along with GHCR publishing.
- **GH Actions job that runs `docker build` in CI.** `make ci` stays host-only; Docker build validated manually per the test plan.

## Tests

1. **Manual — dashboard path.** `make docker-build && mkdir -p data config && cp config.example.yaml config/config.yaml` (edit the DB path to `/data/startup_radar.db`), then `make docker-run` — Streamlit boots at http://localhost:8501, config is read from `/config/config.yaml` via the baked-in `--config`, DB initializes at `/data/startup_radar.db` (verified by `ls -la data/` on the host).
2. **Manual — one-shot pipeline path.** `docker run --rm -v $PWD/data:/data -v $PWD/config:/config startup-radar:phase-14 --config /config/config.yaml run` — container exits 0, `data/startup_radar.db` has a new row in `runs` with `status = 'ok'`.
3. **Manual — secrets not baked.** `docker run --rm --entrypoint bash startup-radar:phase-14 -c "find / -name 'credentials.json' -o -name 'token.json' -o -name '.env' 2>/dev/null"` — returns empty.
4. **Manual — non-root.** `docker run --rm --entrypoint id startup-radar:phase-14` prints `uid=10001(radar) gid=10001(radar)`.
5. **Manual — image size.** `docker images startup-radar:phase-14 --format '{{.Size}}'` ≤ 250 MB (soft target).
6. **Manual — compose validates.** `docker compose config` parses `docker-compose.yml` without errors.
7. **Automated — `--config` flag.** New case in `tests/unit/test_cli_config_flag.py` (new file): invoke the Typer app via `CliRunner` with `--config <tmp_yaml> status`, assert the tmp-path config was loaded (not the package default). No Docker required.
8. **Automated — serve subprocess hand-off.** Second case in the same test: set `STARTUP_RADAR_CONFIG_PATH=<tmp_yaml>` in the environment, call `load_config()` with no path argument, assert it resolves to the tmp-path. Documents the one controlled env var.
9. **Automated — `make ci` stays green.** No regression in lint/mypy/pytest.
10. **No GH Actions Docker build step added.** `grep -rn 'docker build\|docker buildx' .github/workflows/` returns zero matches.
